# 電子卡片專案 — 第十一輪修改規格書

> 此文件為前十份規格書(v1 ~ v10)的補充修正,請一併參考。
> 本輪處理三件事:① 新增「收件人專屬連結」,讓收件人有獨立於訪客/管理者的
> 專屬體驗 ② 訪客連結的「新增留言」按鈕調整為只在留言牆區塊出現,不佔用封面
> ③ 新增「下載卡片存檔(PNG)」功能

---

## 1. 新增「收件人專屬連結」

### 1.1 設計目的
目前訪客連結同時被「收禮的人」跟「留言的親友」共用,導致收禮人打開連結時,
畫面上出現「新增留言」這種功能性按鈕,削弱了「這是專屬於我的一份禮物」的
感受。本輪新增第三種連結,讓收件人有獨立、乾淨、只看不互動(但可下載存檔)
的專屬體驗。

### 1.2 三組連結最終分工

| 連結類型 | 使用對象 | 可以做什麼 |
|---|---|---|
| 訪客連結 `?id=xxx` | 留言的親友 | 看卡片、留言、上傳照片 |
| **收件人專屬連結** `?id=xxx&recipient=yyy`(新增) | 收禮的人 | 看完整卡片(封面動畫、特效、
所有留言拼貼牆),**不顯示新增留言功能**,可下載卡片存檔(PNG) |
| 管理連結 `?id=xxx&admin=yyy` | 建立者 | 編輯/刪除留言、編輯卡片設定、裝飾、換排列方式 |

### 1.3 資料庫異動

```sql
ALTER TABLE ecard_cards
ADD COLUMN recipient_key UUID DEFAULT gen_random_uuid();

-- recipient_key 不是敏感機密資料(外流最多只是讓陌生人也能用「收件人模式」
-- 檢視卡片,不像 admin_key 有編輯/刪除風險),可以安全地加進公開視圖，
-- 供前端比對是否為收件人專屬連結：
DROP VIEW IF EXISTS ecard_cards_public;
CREATE VIEW ecard_cards_public AS
SELECT id, category, style, recipient_name, cover_photo_url,
       blessing_message, show_blessing, creator_signature,
       show_signature, created_at, color_theme, color_adjust,
       decorations, layout_style, recipient_key
FROM ecard_cards;

GRANT SELECT ON ecard_cards_public TO anon;
```

（請依實際 `ecard_cards_public` 視圖目前的完整欄位清單為準調整，
上述僅列出目前已知欄位，若中間輪次有其他欄位請一併保留）

### 1.4 前端呈現邏輯
- 網址帶有正確的 `recipient` 參數(比對 `recipient_key`)時,進入「收件人模式」:
  完整顯示封面(含進場動畫、飄浮特效)、留言拼貼牆(含所有貼紙裝飾),
  但**不顯示「新增留言」按鈕**,改為顯示「下載卡片存檔」按鈕(見第 3 節)
- 網址沒有 `recipient` 參數(一般訪客連結)時,維持現有行為:可以留言、
  上傳照片,不顯示下載存檔按鈕
- 建立卡片完成、產生連結的畫面,需同時顯示**三組連結**,並清楚標示各自用途
  與適合分享的對象(例如加上簡短說明文字,避免建立者搞混哪組連結該給誰)

## 2.「新增留言」按鈕位置調整(適用訪客連結)

訪客連結(給留言親友使用的那組)的「新增留言」按鈕,調整為**只在留言拼貼牆
區塊捲動範圍內才出現**,封面畫面(收件人姓名、照片、祝福語)不顯示任何
功能性按鈕,維持第一眼的乾淨與專屬感受。

---

## 3. 下載卡片存檔(PNG)功能

### 3.1 功能說明
在收件人專屬連結畫面提供「下載卡片存檔」按鈕,點擊後將當下整張卡片
(封面 + 目前為止所有留言與照片 + 所有裝飾貼紙)完整截圖,輸出成單一
PNG 圖檔供下載保存。

### 3.2 技術做法
- 建議使用 `html2canvas`(或專案中已有慣用的類似截圖函式庫)將整個卡片
  容器(封面 + 留言拼貼牆)一次性擷取成圖片,不需要做成 PDF(比照過去
  五行分析、命理工具的經驗,PDF 容易有分頁瑕疵,PNG 是更穩定的做法)
- 因為卡片內容是動態累積的(今天下載跟之後留言更多了再下載,內容會不同),
  下載的圖片僅反映「下載當下那一刻」的樣子,請在下載按鈕附近加上簡短
  提示文字說明這個限制(例如:「將會存下目前顯示的內容」)
- 若卡片留言/照片數量較多,截圖生成可能需要一點時間,請加上讀取中的
  提示動畫,避免使用者以為沒反應而重複點擊
- 生成的圖片檔名建議包含收件人姓名(例如 `給bibi的卡片.png`),方便使用者
  辨識保存的檔案

### 3.3 效能與相容性
- 若留言與照片數量非常多導致單張截圖過於龐大(長度過長、超過瀏覽器
  處理極限),可考慮設定合理的上限,或至少確保發生問題時有清楚的錯誤
  提示,而非直接讓功能卡死無反應
