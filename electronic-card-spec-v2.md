# 電子卡片專案 — 第二輪修改規格書

> 此文件為 electronic-card-spec.md 的補充修正,請兩份文件一起參考。
> 本輪處理三件事:① 照片上傳 bug 除錯 ② 帳號系統與卡片列表 ③ 視覺重新設計(溫馨手作感)
> 三者可依技術相依性自行安排實作順序,但完成後三者都要能一起正常運作。

---

## 0. 前置動作

正式環境目前有 2 筆測試卡片(收件人「小美」、「小华」),請先到 Supabase Table Editor
刪除 `ecard_cards` 裡這兩筆資料(外鍵 `ON DELETE CASCADE` 會連帶清掉對應留言),
清乾淨後再開始本輪開發,避免跟後續帳號系統的資料關聯產生混淆。

## 1. 照片上傳 Bug 除錯

**現象**:留言表單選擇照片後,按下上傳/送出按鈕,畫面完全沒有反應,沒有進度提示、
沒有錯誤訊息、也沒有任何跳動或載入動畫,像沒點到按鈕一樣。

**除錯方向建議**(非確定原因,請實際排查):
- 檢查按鈕的 onClick / onSubmit 事件是否有正確綁定,是否有 JS 錯誤在 console 被吞掉
- 檢查檔案選擇後的 state 是否有正確更新(可能是選完照片但 state 沒存到,導致送出時沒有檔案可傳)
- 檢查是否因為前面 Storage RLS policy 曾經卡過,前端有沒有殘留錯誤的 catch 邏輯只是「靜默失敗」沒有顯示錯誤
- 用瀏覽器開發者工具的 Network 分頁,實際測試一次上傳,確認有沒有發出請求、
  請求回傳的狀態碼與內容是什麼

**要求**:無論根本原因是什麼,修好後必須確保:
- 上傳中要有明確的載入提示(轉圈動畫或進度顯示),不能讓使用者以為沒反應
- 上傳失敗時要有清楚的錯誤提示文字(不需要技術術語,例如「照片上傳失敗,請再試一次」)
- 上傳成功後照片要能正確顯示在對應位置(封面或留言卡片上)

## 2. 帳號系統與跨裝置卡片列表

### 2.1 設計目的
解決「建立卡片後,連結沒保存下來就永遠找不到」的問題,同時一併解決
`electronic-card-spec.md` 完成報告中提到的**限制第 1 項**:
`ecard_cards` 資料表的 `admin_key` 欄位目前對任何人可查詢(`SELECT USING (true)`),
理論上可被外部工具繞過前端直接撈取所有卡片的管理金鑰。加上帳號系統後,
應同步收斂這個查詢權限,只有卡片的建立者本人(登入後)能查到自己卡片的 `admin_key`。

### 2.2 登入方式
使用 Supabase Auth 內建的 **Email Magic Link**(輸入 Email、收信、點連結即登入),
不使用密碼,降低操作門檻。

### 2.3 使用範圍(重要:誰需要登入、誰不需要)
- **建立卡片的人**:需要登入(登入時機可放在建立卡片流程的第一步,或完成卡片精靈後、
  產生連結前,擇一即可,但整個「建立」動作必須綁定登入帳號)
- **留言、上傳照片的訪客**:完全不需要登入、不需要註冊,維持原本零門檻的體驗,
  這點不能被本次修改影響到
- **管理連結(帶 admin 參數)的使用**:討論後決定沿用原本 `admin_key` 比對機制即可,
  不強制要求管理連結的開啟也要登入,因為建立者可能會用不同裝置、不同瀏覽器開啟管理連結,
  這種情境下能用連結直接進入還是比較方便

### 2.4 資料庫異動

```sql
-- 幫 ecard_cards 加上建立者欄位,關聯到 Supabase Auth 的使用者
ALTER TABLE ecard_cards
ADD COLUMN creator_id UUID REFERENCES auth.users(id);

-- 建立卡片時，creator_id 必須是當前登入者的 id，之後才允許新增
-- 移除原本過於寬鬆的 SELECT policy，改為區分「一般查詢」與「本人查詢」兩種情境

DROP POLICY IF EXISTS "任何人可讀取卡片" ON ecard_cards;

-- 給訪客與留言頁用的公開查詢視圖，不包含 admin_key 與 creator_id
CREATE VIEW ecard_cards_public AS
SELECT id, category, style, recipient_name, cover_photo_url,
       blessing_message, show_blessing, creator_signature,
       show_signature, created_at
FROM ecard_cards;

-- 公開視圖任何人可讀
GRANT SELECT ON ecard_cards_public TO anon;

-- 原始資料表只有本人（建立者）能查到完整內容，包含 admin_key
CREATE POLICY "本人可讀取自己的卡片完整資料" ON ecard_cards
FOR SELECT USING (auth.uid() = creator_id);

-- 建立卡片時，creator_id 必須等於當前登入者
DROP POLICY IF EXISTS "任何人可建立卡片" ON ecard_cards;
CREATE POLICY "登入者可建立卡片" ON ecard_cards
FOR INSERT WITH CHECK (auth.uid() = creator_id);
```

**注意**:前端所有「訪客檢視卡片」「留言頁讀取卡片資訊」的查詢,都必須改成查詢
`ecard_cards_public` 這個視圖,而不是原本的 `ecard_cards` 資料表,否則會因權限不足而讀不到資料。
只有「我的卡片列表」頁面(登入後才看得到)才查詢原本的 `ecard_cards` 資料表。

管理連結(帶 `admin` 參數)驗證 `admin_key` 時,因為訪客不一定有登入,這段驗證邏輯建議
改寫成一支 `SECURITY DEFINER` 的資料庫函式(比照留言編輯/刪除的做法),
輸入卡片 id 與 admin_key,回傳是否驗證通過與必要的卡片資料,
不要讓前端直接查詢原表比對,才不會又繞回權限漏洞。

### 2.5 新增頁面

- **登入頁**:簡單的 Email 輸入框 + 送出按鈕,送出後提示「請check信箱點擊連結登入」
- **我的卡片頁**(`/my-cards`,需登入):以卡片列表方式呈現該帳號建立過的所有卡片
  (顯示收件人姓名、類別、建立時間、留言數),每張卡片提供「查看」(進入管理連結畫面)
  與「複製訪客連結」兩個操作入口
- 建立卡片精靈完成後的「產生連結頁」維持原樣(顯示兩組連結即可複製),
  但同時該卡片也會自動出現在「我的卡片」列表中,不用特地去記連結

## 3. 視覺重新設計(核心:找回手作卡片的溫馨與密度感)

### 3.1 問題診斷
目前成品的「紙張拼貼風」與「插畫手繪風」兩種風格視覺上幾乎無法分辨,且整體呈現
偏向標準留言板樣式:留言卡為純白直角平行排列、大量留白、無紙張質感、無傾斜角度、
無重疊堆疊感。封面裝飾元素(圖示)過於單薄,視覺重量集中在畫面正中央一小塊。

### 3.2 三種質感風格,要求做出明顯可辨識的差異

**紙張拼貼風 paper_collage**
- 背景使用淡淡牛皮紙或米色紙張紋理(可用 CSS 疊加細緻雜訊紋理或 SVG 紙紋圖案,
  不要求真實照片級紋理,但要能感覺到「不是純色塊」)
- 留言卡片本身也帶紙張質感(略帶米黃色調、邊緣可做輕微不規則鋸齒或撕邊效果)
- 每張留言卡右上角或左上角加一小段「膠帶」裝飾(半透明色塊,斜貼在卡片一角,
  帶輕微陰影模擬立體感)
- 照片使用方形白邊相框呈現,如同拍立得貼上去的效果

**水彩渲染風 watercolor**
- 背景使用柔和的水彩暈染色塊(可用 CSS 徑向漸層疊加多層、模糊邊緣模擬暈染)
- 留言卡邊緣為柔邊、非直角,可用 border-radius 加大或 SVG 不規則邊框模擬水彩暈開的輪廓
- 色彩之間可以有輕微透疊感(半透明疊色),避免色塊邊界過於銳利
- 照片邊緣做柔邊羽化(mask 或漸層透明度)處理,融入背景而非硬邊相框

**插畫手繪風 illustration**
- 背景可保持較乾淨的底色,但需搭配手繪線條裝飾元素散佈其中(例如生日類別:手繪氣球、
  彩帶、星星等線條插畫,依類別對應不同裝飾主題)
- 留言卡邊框使用手繪感畫框線條(粗細不均勻的線條效果,而非標準直角邊框)
- 照片外框同樣採用手繪畫框樣式

### 3.3 留言拼貼牆:找回堆疊密度與傾斜感

- 每則留言卡沿用資料庫的 `layout_seed` 決定一個固定的隨機傾斜角度,
  建議範圍擴大到 **-6 度到 6 度**之間(目前疑似角度太小或沒生效,幾乎看不出傾斜)
- 桌機與平板寬度:採用類似 Masonry(瀑布流)或允許輕微重疊堆疊的排版方式,
  卡片間距縮小,營造「貼滿一整面牆」的密集感,而不是目前這種大片留白、
  卡片零星散落在畫面中的稀疏感
- 手機寬度:考量單欄捲動的可讀性,重疊效果可以降低或取消,但傾斜角度與紙感/水彩感/
  手繪感的視覺質感仍要維持,並適度縮小卡片間距,避免看起來鬆散
- 卡片本身的陰影需加深、加大模糊範圍,增加立體浮貼感,不要用目前這種極淺的陰影

### 3.4 封面:增加視覺重量與豐富度

- 目前的線條小圖示過於單薄,請依「類別 + 質感風格」的組合,提供更豐富的裝飾畫面,
  例如生日類別搭配紙張拼貼風,可設計成有蠟燭、緞帶、紙花等手繪或紙感元素環繞姓名文字,
  而不是畫面正中央一個孤立的小圖示
- 若使用者有上傳封面照片,相框效果依 2.2 節(既有規格文件)所述的三種風格分別呈現,
  且照片本身可以放大占比,讓封面有真正的視覺主角,而非目前小尺寸置中的呈現方式
- 封面整體背景也要套用對應質感風格的紋理/暈染/插畫裝飾,而不是目前這種近乎純色的背景
