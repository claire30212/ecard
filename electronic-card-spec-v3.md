## 1. 修復權限回歸

### 1.1 留言送出 403 Forbidden
第二輪修改帳號系統時的 RLS 調整,推測不慎影響了 `ecard_messages` 資料表原本
「任何人都能新增留言」的權限(這是 v2 規格書 2.4 節從未要求變動的部分)。
請檢查：

- `ecard_messages` 資料表是否仍保有 `任何人可新增留言`(INSERT, WITH CHECK true)
  與 `任何人可讀取留言`(SELECT, USING true)這兩條政策,且 `anon` 角色是否仍有
  `GRANT SELECT, INSERT ON ecard_messages`(這條 GRANT 在第一輪就設定過,
  請確認沒有被覆蓋或移除)
- 若權限確認都存在但仍是 403,請檢查前端呼叫留言新增 API 時,是否誤用了
  需要登入身份的 client(例如帶了認證 token 但該 token 對應的角色權限不足),
  訪客留言的請求應該用 `anon` 身份呼叫,不需要登入

### 1.2 訪客頁面查詢 403 Forbidden
v2 規格書已要求「訪客檢視卡片、留言頁讀取卡片資訊,都必須改查詢 `ecard_cards_public`
視圖,而非原始 `ecard_cards` 資料表」,但實測顯示仍有請求在查詢原表並被擋下。
請全面檢查前端程式碼,找出所有還在查詢 `ecard_cards`(而非 `ecard_cards_public`)
的地方,並確認：

- 卡片檢視頁(訪客開啟 `?id=xxx`,不含 admin 參數時)的所有查詢都改用
  `ecard_cards_public`
- 只有「我的卡片」列表頁(登入後)、以及管理連結驗證用的 `SECURITY DEFINER` 函式
  內部,才需要查詢或存取原始 `ecard_cards` 資料表

### 1.3 照片載入 400 錯誤
請追查這筆載入失敗的圖片網址的實際問題（檔名編碼、bucket 路徑、或該筆資料庫紀錄
儲存的 `photo_url` 本身是否正確），並確認資料庫裡目前所有卡片/留言的 `photo_url`、
`cover_photo_url` 欄位存的網址都能正常開啟。若在檢查過程中發現過去測試資料
（例如小美、小华、ccc 等測試卡片)有累積無效的圖片網址紀錄，請一併清除。

## 2. 登入方式改為帳號密碼制

拿掉 Magic Link 方式，改為傳統的 Email + 密碼 註冊／登入：

- 新增「註冊」畫面：Email + 密碼（+ 密碼確認）
- 新增「登入」畫面：Email + 密碼
- 沿用 Supabase Auth 內建的 Email/Password 登入方式即可，不需要額外服務
- 密碼輸入框需有基本強度提示（例如至少 6 碼），但不需要過度複雜的規則，
  維持操作單純
- 忘記密碼可先不做重設功能（超出本輪範圍），若使用者忘記密碼，
  暫時只能請他們重新用同一個 Email 註冊一個新帳號這種陽春處理方式即可，
  之後有需要再另外評估加上「忘記密碼」流程

## 3. 內建插畫選項擴充

每個類別（生日 / 紀念日 / 祝福 / 感謝）至少提供 **5 張風格不同**的內建插畫供選擇，
數量不設上限，由 Claude Code 自行判斷合理範圍即可（例如 5 到 8 張皆可）。

- 插畫內容需與該類別主題相關（例如生日類別可以有蛋糕、氣球、蠟燭、禮物盒、
  彩帶等不同構圖方向的插畫，而非同一張圖的顏色變體）
- 建立卡片精靈選擇「內建插畫」時，需以縮圖清單或輪播方式呈現這些選項，
  讓使用者可以清楚比較後再選定一張
- 插畫本身的線條/色彩風格，需能合理套用先前規劃的三種質感風格
  （紙張拼貼／水彩渲染／插畫手繪）對應的封面裝飾效果，不需要每張插畫都做
  三種版本，但整體呈現不能與所選質感風格衝突過於突兀
